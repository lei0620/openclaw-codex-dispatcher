# NAS Codex 手机遥控稳定版设计

## 1. 背景

当前项目已经具备手机面板、NAS Dispatcher、Win11 代理和 Codex 会话调用能力，但真实使用仍存在以下问题：

- 手机能连接 NAS，但“电脑在线”“Codex 可用”“绑定窗口在线”混成一个状态，容易误判。
- 手机会话与桌面可见窗口曾通过输入模拟耦合，切换会话时可能发错位置。
- Codex 更新后桌面主进程从 `Codex.exe` 变为 `ChatGPT.exe`，窗口识别失效。
- Codex app-server 在第一条任务到来时才启动，首条消息存在启动延迟和失败窗口。
- 手机主要依赖轮询，消息、权限和执行状态不能稳定实时到达。
- 重连、重复提交、离线恢复和并行会话缺少统一协议。
- Android 调试日志可能包含访问凭据。

本设计采用用户确认的方案：保留飞牛 NAS 作为专用中转，但运行链路不依赖 OpenClaw。

## 2. 目标

手机 App 应成为 Codex 的可靠远程客户端，在同一局域网内实现：

1. 查看 `D:\aixm` 下已发现的项目和最近会话。
2. 新建、切换和继续真实 Codex 会话，不向错误窗口发送内容。
3. 实时接收用户消息确认、流式回复、任务状态和权限请求。
4. 同时控制不同会话并行工作，同一会话严格串行。
5. 手机、NAS、Win11 或 Codex 短暂断线后自动恢复，不丢消息、不重复执行。
6. Win11 登录并打开 Codex 后自动进入可用状态，不需要手动启动脚本。
7. 在手机上清楚区分 NAS、电脑代理、Codex 会话服务和桌面显示同步的状态。
8. 保留项目白名单、权限确认和高风险操作限制。

## 3. 非目标

- 第一阶段不做通用 Windows 桌面自动化。
- 第一阶段不支持多台 Win11 调度。
- 第一阶段只保证局域网访问；外网访问继续由 Tailscale/VPN 作为独立能力处理。
- 不把剪贴板、鼠标点击或前台窗口识别作为任务执行主通道。
- 不承诺依赖未公开的 Codex 桌面深链。桌面画面跟随失败时，任务和会话仍必须正确执行。

## 4. 总体架构

```text
Android App
  |  HTTPS/REST: 登录、初始快照、历史分页
  |  WebSocket: 实时事件、消息确认、权限处理
  v
NAS Relay
  |  持久任务队列、事件游标、幂等键、在线状态
  |  单一出站 WebSocket
  v
Win11 Agent
  |  app-server 监督器、项目发现、会话同步、权限桥接
  v
Codex app-server (127.0.0.1 only)
  |  thread/start, thread/resume, turn/start, turn events
  v
Codex local state / desktop app
```

### 4.1 稳定性原则

- `codexSessionId` 是会话身份的唯一依据，窗口句柄不是会话身份。
- NAS 是消息确认、排队和重连游标的唯一中转真相源。
- Win11 代理只主动连接 NAS，不在局域网开放控制端口。
- Codex app-server 仅监听 `127.0.0.1`，由监督器保持可用。
- 桌面输入模拟默认关闭，只保留为显式诊断工具。
- 桌面画面跟随属于显示层，失败不能把已成功的 Codex turn 标记为失败。

## 5. 组件设计

### 5.1 NAS Relay

NAS Relay 由现有 Dispatcher 演进而来，不依赖 OpenClaw。它负责：

- 手机认证和 Win11 代理认证使用不同凭据。
- 保存项目、会话映射、任务、权限请求和有限事件历史。
- 为每条手机消息生成或接收稳定的 `clientMessageId`。
- 同一个 `clientMessageId` 只允许创建一次任务。
- 每个会话最多运行一个 turn；不同会话可以并行。
- 代理离线时任务保持 `queued`，重连后继续分配。
- 保存递增事件序号，客户端用 `lastEventId` 恢复遗漏事件。
- 维护 NAS、代理和 Codex 服务三层健康状态。

### 5.2 Win11 Agent

代理通过 Windows 登录任务自动启动，并持续运行，不依赖 Codex 窗口是否处于前台。它负责：

- 连接 NAS，发送 10 秒心跳和能力快照。
- 发现 `D:\aixm` 项目及最近 Codex 会话。
- 监督本地 app-server；不可用时自动启动，退出后自动重启。
- 把任务准确映射到 `thread/start` 或 `thread/resume`，再执行 `turn/start`。
- 实时回传 turn 事件、最终结果、失败原因和权限请求。
- 根据会话维度排队，允许不同会话并行。
- 兼容 Codex 桌面宿主进程 `ChatGPT.exe`、旧版 `Codex.exe` 以及后续可配置匹配规则。
- 将窗口刷新/显示跟随作为独立操作，不参与消息路由。

### 5.3 Codex app-server 监督器

当前本机 Codex CLI 将 `app-server` 标记为 experimental，因此实现必须隔离版本变化：

- 启动前读取本机最新 Codex 可执行文件并验证 `app-server --help`。
- 默认使用 loopback WebSocket，不监听局域网地址。
- 代理启动后立即预热，而不是等待第一条手机消息。
- `readyz` 连续失败或进程退出时按 1、2、5、10、30 秒退避重启。
- 对协议方法和事件做单独适配层，业务层不直接解析原始消息。
- 启动失败时手机明确显示“Codex 服务不可用”，不显示成“电脑离线”。
- 升级 Codex 后执行兼容性自检，并在诊断页显示版本和协议状态。

### 5.4 Android App

Android App 保留 Capacitor，但连接与凭据由原生插件处理：

- REST 只用于登录、初始快照、历史分页和更新检查。
- 实时消息统一使用原生 WebSocket，支持锁屏/切后台后重连。
- 访问凭据保存到 Android Keystore 支持的安全存储，不进入 JS 日志。
- 每条发送消息先显示本地气泡，并经历 `发送中 -> 已接收 -> 执行中 -> 完成/失败`。
- 收到 NAS 确认后按 `clientMessageId` 合并，不创建重复气泡。
- 切换会话只改变当前 `conversationId`；发送时始终显式携带该 ID。
- App 恢复前台后先用事件游标补齐，再刷新当前会话，避免跳动和重复。

## 6. 会话和消息模型

### 6.1 会话

每个会话记录：

- `conversationId`: NAS 内稳定 ID。
- `projectId`: 所属项目。
- `codexSessionId`: 对应 Codex thread，创建前可为空。
- `title`: 用户可修改标题。
- `revision`: 单调递增版本号。
- `lastEventId`: 最近同步事件。
- `displayWindowId`: 可选，仅用于桌面画面跟随。

新建手机会话的第一条消息由 app-server 创建真实 thread，NAS 随即保存返回的 `codexSessionId`。已有会话必须通过该 ID 恢复，禁止退回当前前台窗口。

### 6.2 消息发送流程

```text
手机创建 clientMessageId
  -> NAS 幂等接收并返回 accepted
  -> 会话队列分配给 Win11
  -> Win11 恢复/创建 Codex thread
  -> Codex 开始 turn
  -> 流式事件经 NAS 推送手机
  -> 完成事件写入 NAS
  -> 手机按 clientMessageId/turnId 合并最终状态
```

超时只表示客户端暂时没收到结果，不代表可以自动重复执行。重连后必须先查询同一 `clientMessageId` 的状态。

## 7. 实时事件协议

手机 WebSocket 至少支持：

- `connection.snapshot`
- `agent.status`
- `codex.status`
- `conversation.created`
- `conversation.updated`
- `message.accepted`
- `turn.started`
- `turn.delta`
- `turn.completed`
- `turn.failed`
- `approval.requested`
- `approval.resolved`
- `task.cancelled`
- `sync.required`

每个事件包含 `eventId`、`occurredAt`、`conversationId` 和必要的幂等标识。手机确认最后处理的事件，NAS 仅保留有限窗口并在游标过旧时要求完整同步。

## 8. 掉线与恢复

- 手机 WebSocket 断开后按指数退避自动重连，最大间隔 30 秒并加入随机抖动。
- 代理 10 秒心跳；30 秒未收到心跳标记为离线。
- 代理断开时，运行中的任务进入“结果待确认”，不立刻重新执行。
- 代理重连后先上报仍在运行的 turn，再由 NAS 对账。
- app-server 重启后，未完成 turn 先查询 thread 状态；无法确认时提示用户重试，禁止静默重复执行。
- App 切后台期间的新事件通过游标补齐；权限请求同时触发系统通知。

## 9. 手机交互设计

### 9.1 顶部状态

顶部只显示一个组合状态入口，图标颜色表示：

- 绿色：NAS、电脑、Codex 均可用。
- 黄色：NAS 可用，但电脑或 Codex 正在恢复。
- 红色：NAS 不可达。

点击后展开三个独立状态及最近心跳时间。窗口绑定不再显示成电脑“离线”。

### 9.2 会话列表

- 按项目组织，只预载每个项目最近 3 个会话。
- 当前运行、等待权限、失败状态使用小型状态标记。
- 新建会话立即出现在当前项目顶部。
- 切换会话不改变其他会话的执行状态。
- 顶部提供“正在执行”入口，集中查看并切换所有活跃会话。

### 9.3 聊天页

- 顶栏固定，输入区固定在安全区域底部。
- 用户消息使用桌面端相近的浅灰气泡；Codex 回复使用无大卡片的正文布局。
- 流式回复在同一消息节点增量更新，不重复创建临时消息。
- 用户向上浏览历史时不自动跳到底部；新消息以“回到最新”按钮提醒。
- 发送、停止、重新发送和权限确认都作用于当前会话。

### 9.4 设置与诊断

设置页包含服务地址、凭据更新、检查更新和通知开关。诊断页显示：

- NAS 延迟和版本。
- 代理在线时间、版本和最近心跳。
- Codex 版本、app-server ready 状态和最近错误。
- 当前会话 ID 与 Codex thread ID。
- 一键导出脱敏诊断信息。

## 10. 安全设计

- 手机凭据与代理凭据分离并支持轮换。
- Android 凭据不写入 WebView localStorage、日志或崩溃报告。
- app-server 仅监听 loopback。
- NAS 接口继续校验项目白名单和任务来源。
- 权限请求必须绑定 task、conversation 和 agent，过期后不可批准。
- 所有日志在上传或导出前移除 token、Authorization 和用户密码。

## 11. 分阶段实施

### 阶段 A：连接核心

- 修复新版 `ChatGPT.exe` 窗口识别，并将窗口状态与代理状态分离。
- 增加 app-server 监督器、预热、ready 状态和自动重启。
- 完成手机、NAS、代理三层健康状态。
- 禁止会话任务回退到桌面输入模拟。

### 阶段 B：可靠实时消息

- 新增手机 WebSocket 事件通道和事件游标。
- 新增 `clientMessageId` 幂等、确认状态和重连对账。
- 完成同会话串行、不同会话并行。
- 完成权限请求实时推送和通知。

### 阶段 C：手机体验

- 重构状态入口、会话列表、活跃会话和聊天消息状态。
- 修复滚动、输入框、安全区域、流式显示和重复消息。
- 增加诊断页和脱敏导出。

### 阶段 D：安全与发布

- 把凭据迁移到安全存储并清除调试日志泄露。
- 自动化 Win11 登录启动、NAS 容器恢复和版本兼容检查。
- 构建 APK、真机安装、WebDAV 更新和回滚包。

## 12. 验收标准

1. Win11 重启并登录后，无需手动命令，手机 30 秒内显示电脑和 Codex 可用。
2. 连续切换 5 个会话各发送消息，全部进入对应 Codex thread，无串话。
3. 同时运行 2 个不同会话时可以并行；同一会话的两条消息严格按顺序执行。
4. 连续发送 20 条短消息，手机和电脑会话中不丢失、不重复。
5. 在任务运行中分别断开手机 Wi-Fi、重启 NAS 服务和重启代理，恢复后不自动重复执行。
6. Codex app-server 被结束后 10 秒内开始恢复，手机显示准确恢复状态。
7. 权限请求在局域网正常情况下 2 秒内出现在手机，并能批准或拒绝。
8. 手机向上查看历史时不会被流式消息强制拉到底部。
9. Android 全面屏手势、状态栏和底部安全区域无重叠。
10. Android logcat、NAS 日志和代理日志中不出现完整凭据。
11. 连续运行 24 小时后，在线状态、会话列表和事件游标仍保持一致。

## 13. 测试策略

- 单元测试：窗口识别、会话映射、幂等键、事件游标、任务调度、状态机。
- 集成测试：NAS WebSocket、代理重连、app-server 适配、权限生命周期。
- 故障注入：断网、重复请求、代理退出、app-server 退出、NAS 重启和游标过期。
- 真机测试：Android 前后台切换、锁屏恢复、通知、手势导航、滚动和输入法。
- 端到端测试：至少两个项目、五个会话、两个并行任务和一次权限确认。
- 发布门禁：全量测试、TypeScript 构建、Android 构建、真机截图、日志脱敏检查和回滚验证。

## 14. 设计决策总结

- 保留 NAS 是为了可靠排队、重连和状态对账，不是为了依赖 OpenClaw。
- Codex 会话协议负责正确性；桌面窗口只负责显示跟随。
- 手机体验以实时事件和明确状态为核心，不继续扩大轮询逻辑。
- 对实验性 app-server 建立适配层和监督器，降低 Codex 更新造成的破坏范围。
- 第一阶段先稳定连接核心，再逐步替换消息和界面，避免一次性重写导致不可用。
