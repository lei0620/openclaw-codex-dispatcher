# Codex 桌面与手机会话列表一致性设计

## 背景

手机端当前按每个项目最近 5 条加载 Codex 会话。实际排查发现，Win11 同步器使用 `session_index.jsonl` 的 `updated_at` 排序，但 Codex 桌面端使用 `state_5.sqlite` 中的 `recency_at_ms` 排序，并同时读取 `archived` 状态。

“配置飞牛NAS源”在 `session_index.jsonl` 中仍是 2026-07-04，但在桌面数据库中的最近活跃时间是 2026-07-13。因此它在桌面端排第 2，在手机同步结果中排第 6，最终被 5 条限制截掉。

当前同步器还会每 2.5 秒扫描并解析最多 150 个会话文件。实机测量一次同步约需 3.5 秒，会持续占用 Win11 代理事件循环，影响心跳、窗口同步和消息响应。

## 目标

- 手机每个项目仍只显示最近 5 个 Codex 对话。
- 最近顺序与 Codex 桌面端一致。
- 已归档会话不参与最近 5 条选择。
- Codex 内部子智能体、审查和工作线程不进入手机普通会话列表。
- 保留 `session_index.jsonl` 中用户可见的自定义会话名称。
- 只解析最终入选会话的消息文件，降低同步耗时。
- Codex 数据库不可用时自动回退到现有文件扫描逻辑，不影响代理启动。

## 方案

Win11 同步器新增只读桌面线程元数据读取：

1. 从 `$CODEX_HOME/state_5.sqlite` 读取 `id`、`cwd`、`archived`、`recency_at_ms` 和 `updated_at_ms`。
2. 过滤 `archived = 1` 的线程，以及 `thread_source = 'subagent'` 的内部线程；兼容旧数据库中 `thread_source` 为空的用户线程。
3. 根据线程 `cwd` 匹配白名单项目，并按 `recency_at_ms` 选择每项目前 5 条。
4. 从 `session_index.jsonl` 读取用户在桌面端看到的会话名称。
5. 只查找和解析入选线程对应的 rollout 文件，继续保留最近三轮消息和现有文本清理规则。
6. 如果数据库缺失、版本不兼容或查询失败，使用原有按会话文件更新时间排序的兼容路径。

NAS 将 Win11 上报视为指定项目的完整最近会话快照：

1. 同步消息同时携带本次覆盖的项目 ID，即使某个项目当前没有入选会话也能正确对账。
2. NAS 删除覆盖项目内未出现在新快照中的旧 Codex 会话，避免历史归档或内部线程继续占位。
3. 已经关联手机任务的会话不会被快照清理，以保留手机发送记录、结果和未读状态。
4. 被清理的会话通过 `conversation.deleted` 实时事件通知手机，无需手动刷新。

SQLite 访问使用 Node.js 22 自带的只读 `node:sqlite`，不引入需要编译的第三方原生依赖，也不会修改 Codex 数据库。

## 不变范围

- 不改变手机发送任务、窗口绑定、同窗口排队和权限处理逻辑。
- 不改变 NAS、Tailscale/VPN 和访问密码配置。
- 不加载全部历史会话，仍遵循每项目最近 5 条的性能边界。
- 不修改 Codex 自身数据库、会话文件或桌面端状态。

## 测试

- 构造桌面数据库中最近活跃、但索引更新时间较旧的会话，确认它仍进入最近 5 条。
- 构造归档会话，确认它不会占用最近 5 条名额。
- 构造 `thread_source = 'subagent'` 的内部线程，确认它不会出现在手机普通会话列表。
- 确认会话标题仍采用 `session_index.jsonl` 的桌面显示名称。
- 删除测试数据库，确认原有文件扫描回退仍工作。
- 运行完整单元测试和 TypeScript 构建。
- 在真实 Win11 数据上对比修复前后标题顺序与同步耗时。
- 触发 NAS 会话同步并在手机真机检查目标对话可见。
- 预置旧 NAS 会话，确认完整快照会清理无任务历史的旧记录，并保留已有手机任务的会话。
- 确认手机收到删除事件后立即移除旧会话，并在当前会话被清理时自动选择有效会话。

## 成功标准

- `D:\aixm\对话` 的手机最近列表与桌面截图一致，包含“配置飞牛NAS源”。
- 同步器不再把归档会话放进最近列表。
- 单次真实会话同步耗时显著低于当前约 3.5 秒，并小于 2.5 秒同步周期。
- 代理心跳、实时消息和窗口同步测试无回归。
